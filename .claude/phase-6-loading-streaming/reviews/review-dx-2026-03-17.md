## Findings

### Finding 1: setStreaming called on every status transition including non-streaming ones [MED]

**Location:** `web/src/pages/WorkspaceUseChat.tsx:132‚Äì134`

**Issue:**
The `useEffect` fires on every `status` change ‚Äî including `'ready'`, `'error'`, `'submitted'`, and `'streaming'`. The effect only writes `true` when `status === 'streaming'` and `false` otherwise. This means a status of `'submitted'` (user has sent, model has not yet started streaming) will set `isStreaming = false` in the store, which correctly suppresses the green dot ‚Äî but the intent might be to show the dot during `'submitted'` too (the header already shows a "Sending‚Ä¶" badge for `'submitted'`). The decision is arguably fine as-is, but it is implicit: a future developer reading `conn.isStreaming` in the Sidebar won't know whether `'submitted'` was intentionally excluded.

**Evidence:**
```typescript
useEffect(() => {
  setStreaming(sessionId, status === 'streaming')
}, [sessionId, status, setStreaming])
```

**Developer Impact:**
- Affects: any future developer adding a status indicator that needs to distinguish `submitted` from `streaming`
- Time wasted: ~15‚Äì30 minutes of debugging why the dot doesn't appear during the "Sending‚Ä¶" phase
- Frequency: would surface whenever the indicator semantics are revisited

**Fix (comment only, no code change required):**
```typescript
// NOTE: 'submitted' (user sent, model not yet streaming) is intentionally excluded ‚Äî
// the header badge handles that state. The sidebar dot only shows during active streaming.
useEffect(() => {
  setStreaming(sessionId, status === 'streaming')
}, [sessionId, status, setStreaming])
```

**Implementation:**
1. Add the one-line comment above the `useEffect` to make the exclusion intentional and searchable.

### Finding 3: Streaming dot uses a custom animation string instead of a named Tailwind utility [LOW]

**Location:** `web/src/components/layout/Sidebar.tsx:157`

**Issue:**
The streaming dot uses `animate-[pulse_0.75s_ease-in-out_infinite]` ‚Äî an arbitrary Tailwind value ‚Äî while the unread dot uses `animate-pulse`. If the design token for "streaming animation" changes in the future, this is a one-off string that won't be caught by a search for `animate-pulse`.

**Evidence:**
```tsx
<span className="w-2 h-2 rounded-full bg-success animate-[pulse_0.75s_ease-in-out_infinite]" title="Streaming" />
```

**Developer Impact:**
- Affects: any designer or developer modifying animation tokens
- Time wasted: ~5 minutes spotting the discrepancy
- Frequency: low (animation tokens rarely change)

**Fix (either approach):**
Option A ‚Äî add `animate-streaming` to `tailwind.config.ts`:
```typescript
extend: {
  animation: {
    streaming: 'pulse 0.75s ease-in-out infinite',
  }
}
```
Then use `animate-streaming` in the JSX.

Option B ‚Äî leave as-is and add a TODO comment for the next Tailwind config pass.

### Finding 4: `setStreaming` prop is threaded one extra component layer unnecessarily [NIT]

**Location:** `web/src/pages/WorkspaceUseChat.tsx:59, 82, 93‚Äì106`

**Issue:**
`setStreaming` is selected from the store in `WorkspaceChatView`, passed as a prop to `WorkspaceChatSessionReady`, and consumed there. Because `WorkspaceChatSessionReady` is already inside the React tree with access to `useSessionsStore`, it could select `setStreaming` itself and eliminate the prop. Prop-threading stable store selectors is minor friction for code readers.

**Developer Impact:**
- Affects: developers reading `WorkspaceChatSessionReady`'s prop signature
- Time wasted: ~2 minutes tracing whether the prop adds extra reactivity

**Fix:**
```typescript
// Inside WorkspaceChatSessionReady ‚Äî remove setStreaming from props
const setStreaming = useSessionsStore((state) => state.setStreaming)
```
Then remove it from `WorkspaceChatView`'s selection and from the prop interface.
