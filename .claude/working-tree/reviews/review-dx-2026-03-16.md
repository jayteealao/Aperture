## Summary

Phase 4 is a well-executed refactor that replaces ~210 lines of custom input/attachment handling with a declarative, composable `<PromptInput>` component tree. The migration significantly improves the developer experience for building and maintaining chat input UIs: attachment validation, paste handling, keyboard shortcuts, and blob-to-data-URL conversion are now encapsulated in reusable primitives rather than scattered across the page component. The composable API (`PromptInputHeader`, `PromptInputBody`, `PromptInputFooter`, `PromptInputSubmit`) is well-structured and discoverable.

The main DX concern is the large surface area of the `prompt-input.tsx` module (1464 lines, ~40 exports) which presents a steep learning curve for contributors unfamiliar with the component library. No tests were added or modified in these commits, which is a gap given the behavioral changes to submit/error/retry flow.

## Findings

### Finding 1: No tests for new behavioral contracts [HIGH]

**Location:** `web/src/pages/WorkspaceUseChat.tsx`, `web/src/components/chat/AttachmentsPreview.tsx`

**Issue:**
Phase 4 changes the submit flow significantly: the old `handleSend` managed `input` state, `attachedImages` state, and FileReader conversion inline. The new `handleSubmit` delegates to PromptInput's blob-to-data-URL pipeline and relies on throw-to-preserve-input error semantics. No test covers:
- That `handleSubmit` throws on disconnection (preserving input)
- That `handleSubmit` re-throws sendMessage errors (preserving input)
- That `handleFileError` surfaces toast messages
- That `AttachmentsPreview` renders/removes files correctly

**Evidence:**
```typescript
// handleSubmit throws to signal PromptInput to preserve user input
if (!connection || connection.status !== 'connected') {
  toast.error('Not connected', 'Check your connection and try again.')
  throw new Error('Not connected')  // <-- new contract, untested
}
```

**Developer Impact:**
- Future contributors may accidentally break the throw-to-preserve contract without test coverage catching it
- Time wasted: 1-2 hours debugging if retry-on-error regresses
- Affects: all contributors touching chat input
- Frequency: every time submit flow is modified

**Fix:**
Add a test file `web/src/pages/WorkspaceUseChat.test.tsx` (or extend existing) that verifies:
1. `handleSubmit` throws when disconnected
2. `handleSubmit` re-throws on sendMessage failure
3. `AttachmentsPreview` renders thumbnails and handles remove

**Implementation:**
1. Create test file using vitest + @testing-library/react
2. Mock `useChat`, `useSessionsStore`, `useToast`
3. Assert throw behavior and toast calls

### Finding 2: prompt-input.tsx is a 1464-line single file with ~40 exports [MED]

**Location:** `web/src/components/ai-elements/prompt-input.tsx`

**Issue:**
The PromptInput module that Phase 4 now depends on is a single 1464-line file exporting approximately 40 components, hooks, types, and contexts. While the code is well-organized with section comments, a new contributor trying to understand how `AttachmentsPreview` hooks into `usePromptInputAttachments` must navigate through multiple context layers (LocalAttachmentsContext, ProviderAttachmentsContext, PromptInputController) in a single file.

**Developer Impact:**
- New contributors need 15-20 minutes to trace the context hierarchy
- Affects: anyone extending or debugging the prompt input
- Frequency: every prompt input feature change

**Fix:**
Consider splitting into sub-modules:
```
prompt-input/
  index.ts           # re-exports
  PromptInput.tsx     # root component + contexts
  PromptInputTextarea.tsx
  PromptInputSubmit.tsx
  PromptInputActions.tsx  # action menu components
  attachments.ts     # AttachmentsContext, hooks, add/remove logic
  types.ts           # PromptInputMessage, prop types
```

**Implementation:**
1. Create directory structure
2. Move each section (clearly marked with `// ===` comments) into its own file
3. Re-export from index.ts (zero breaking changes for consumers)

### Finding 3: handleFileError code type is looser than source [MED]

**Location:** `web/src/pages/WorkspaceUseChat.tsx:175-179`

**Issue:**
The `handleFileError` callback accepts `{ code: string; message: string }` but PromptInput's `onError` prop emits `{ code: "max_files" | "max_file_size" | "accept"; message: string }`. The declared type `{ code: string }` is looser than the source, meaning a contributor reading only `WorkspaceUseChat.tsx` does not know the possible error codes without checking the PromptInput source.

**Evidence:**
```typescript
const handleFileError = useCallback(
  (err: { code: string; message: string }) => {  // <-- string, not union
    toast.error('File error', err.message)
  },
  [toast]
)
```

**Developer Impact:**
- Minor: contributor adding code-specific handling (e.g., different toast for max_files vs accept) must look up the union type elsewhere
- Time wasted: 5-10 minutes
- Affects: contributors extending error handling
- Frequency: low

**Fix:**
Import the error type from prompt-input or inline the union:
```typescript
import type { PromptInputProps } from '@/components/ai-elements/prompt-input'

type FileError = NonNullable<PromptInputProps['onError']> extends (err: infer E) => void ? E : never
```
Or simply use the literal union directly.

### Finding 4: "Not connected" error could be more actionable [LOW]

**Location:** `web/src/pages/WorkspaceUseChat.tsx:154-157`

**Issue:**
When the user submits while disconnected, the toast says "Not connected / Check your connection and try again." This is slightly vague -- the user cannot reconnect manually. A more actionable message would indicate that the system is attempting to reconnect or suggest refreshing.

**Evidence:**
```typescript
toast.error('Not connected', 'Check your connection and try again.')
throw new Error('Not connected')
```

**Developer Impact:**
- Minor UX friction; users may not know what action to take
- Affects: end users, not developers directly
- Frequency: occasional (network drops)

**Fix:**
```typescript
toast.error('Not connected', 'The WebSocket connection was lost. Reconnection is automatic -- please wait a moment and retry.')
```

---

### Finding 5: No CI test/lint/typecheck pipeline on pull requests (pre-existing) [LOW]

**Location:** `.github/workflows/` (entire directory)

**Issue:**
The only CI workflows are `docker-build.yml` and `build-addon.yml`. There is no workflow that runs `pnpm test`, `pnpm lint`, or `pnpm type-check` on pull requests. This means broken code, type errors, and lint violations can be merged without any automated guard. While pre-existing, the Phase 4 refactor makes this more relevant because the new throw-to-preserve-input contract has no automated verification.

**Developer Impact:**
- Time wasted: 30-60 minutes per broken merge (debugging, reverting, re-fixing)
- Affects: all developers
- Frequency: every PR without manual discipline

**Fix:**
Add a CI workflow that runs on `pull_request` with `pnpm test:all`, `pnpm lint:all`, `pnpm type-check:all`.
